# Atmos

Sitio web clima estilo **"Editorial Atmospheric Full-Bleed"** — pieza portfolio candidatura alternance. Copy interfaz en **francés**.

## Signature element

"Canvas Atmosférico Vivo": foto full-bleed real ciudad seleccionada + overlay gradiente (siempre oscuro abajo → transparente arriba, legibilidad texto claro) cuya hue/opacidad varían según hora local real ciudad (`timezone_offset` / `utc_offset_seconds` de Open-Meteo, 4 buckets: aube/jour/crépuscule/nuit) y condición meteorológica (6 familias: despejado/nubes/lluvia/tormenta/nieve/niebla, mapeadas desde códigos WMO). Temperatura en Fraunces gigante superpuesta directo sobre foto. Crossfade con GSAP al cambiar ciudad. Implementado en `src/domains/hero-temperatura/`.

**Layout tipo Manarola/Tours (desde 2ª iteración diseño)**: header sin selector ciudad (buscador ya cubre eso), índice numerado lateral izquierdo (01 Ressenti / 02 Vent / 03 Humidité / 04 UV, `IndiceLateral.tsx`, labels en `atmos-gold`) distribuido con `justify-between` entre header y borde inferior, bloque temperatura movido a derecha centrado verticalmente (antes abajo-izquierda), indicador "Défiler ↓" fijo abajo-izquierda (`IndicadorScroll.tsx`). Buscador se movió con bloque temperatura, icono lupa ahora a derecha del texto, borde inferior pasa a `atmos-gold` con `focus-within` al enfocar input.

## Stack

- Frontend: React 19 + Vite 8 + Tailwind CSS v4 (config CSS-first vía `@theme` en `src/styles/tokens.css`, sin `tailwind.config.js`)
- Backend: Node.js + Express 5 (proxy API keys, frontend nunca llama directo APIs externas) — ESM, `server/` tiene propio `package.json`
- Clima: **Open-Meteo** (`/v1/forecast`), proxied en `server/services/climaService.js` — sin API key ni tarjeta, gratis hasta 10 000 llamadas/día uso no comercial. Requiere atribución CC BY 4.0 ("Données météo : Open-Meteo.com" con enlace, visible en `DatosDetalle.tsx`).
- Fotos ciudad: **set curado estático** (decisión explícita usuario, no Unsplash API) — 19 ciudades en `src/lib/data/ciudades.ts`, fotos placeholder SVG generadas en `public/images/villes/*.svg` **pendientes reemplazo por fotografía real curada**
- Noticias: mock (`server/services/noticiasService.js`), forma respuesta ya compatible con API real futura
- Animación: GSAP (crossfade fondo en `FondoDinamico.tsx`) + Lenis (smooth scroll, `shared/hooks/useLenis.ts`)
- Data fetching: React Query (`shared/hooks/useClimaActual.ts`)

## Arquitectura

Domain-driven. Cada dominio en `src/domains/<nombre>/` autocontenido: `components/`, `hooks/` (si aplica), archivo principal `<Nombre>.tsx`, `index.ts` exporta solo API pública dominio.

Dominios: `header`, `hero-temperatura`, `forecast`, `datos-detalle`, `noticias`.

Compartido: `src/shared/hooks` (`useClimaActual`, `useLenis` — usados por varios dominios a la vez, por eso viven aquí no dentro de dominio), `src/lib/api` (cliente HTTP + tipos hacia backend), `src/lib/data/ciudades.ts` (set curado), `src/lib/clima/condicion.ts` (mapeo códigos WMO), `src/styles/tokens.css` (design tokens).

`App.tsx` composition root: mantiene estado ciudad seleccionada y único `useClimaActual`, pasa datos hacia abajo a cada dominio por props (evita refetches duplicados y prop-drilling estado global).

Backend en `server/` con `routes/` y `services/` separados por dominio datos (`clima`, `noticias`).

## Diales taste-skill

- **VARIANCE**: 6/10 — identidad distintiva repetible (serif + atmósfera + superposición), sin caos ni experimentación dispersa por sección
- **MOTION**: 4/10 — animación ambiental (overlay/fondo), nada scroll-jacking ni efectos dispersos
- **DENSITY**: 3/10 — whitespace generoso, dato hero dominante, secundarios en grid 3-4 ítems máx

## Trazo animado bajo "Conditions actuelles" — 2026-07-08

`SubrayadoAnimado.tsx`: línea SVG dorada ondulada que se dibuja al entrar la sección en viewport (scroll hacia abajo), lenta y suave, se repite cada vez que se re-entra — a partir de referencia visual que trajo el usuario.

- **Primer intento con GSAP ScrollTrigger (patrón ya usado en `useEncogimientoScroll.ts`) falló**: bug real encontrado muestreando `strokeDashoffset` cada 100ms — en vez de progresar gradual (1→0 en 1.8s), saltaba de golpe en una sola muestra (0 valores intermedios en 5+ lecturas). Causa: `gsap.ticker.lagSmoothing(0)` — activado globalmente en `useLenis.ts` porque el scroll-shrink del hero (con `scrub: true`) necesita reaccionar sin suavizado a cada frame de scroll — hace que cualquier micro-lag del ticker salte tweens de duración fija completos en vez de interpolarlos. Afecta a cualquier tween GSAP de duración fija (no scrub) que se agregue al proyecto de ahora en más, no solo a este.
- **Fix**: reemplazado GSAP por `IntersectionObserver` + transición CSS pura (`transition: stroke-dashoffset 1.8s cubic-bezier(...)`) — el compositor del navegador maneja la interpolación, inmune al ticker de GSAP. `pathLength={1}` normaliza el largo del trazo a 1, así `strokeDasharray`/`strokeDashoffset` no dependen de la geometría real del path SVG.
- **Reverificado con la técnica correcta** (muestreo cada ~100ms): progresión real 1 → 0.888 → 0.607 → 0.408 → ... → 0, curva continua, ~1.8-2s total. Repetición confirmada: al salir de la sección el trazo vuelve hacia oculto, al re-entrar se redibuja desde ahí.
- `rootMargin: '0px 0px -15% 0px'` en el observer aproxima el mismo punto de disparo que tenía `start: 'top 85%'` en la versión GSAP descartada.
- **Fix pedido por usuario**: trazo abarcaba el título entero ("Conditions actuelles"), debía ir solo bajo "actuelles". Reestructurado: `SubrayadoAnimado` ahora envuelve la palabra en un `<span position:relative>` (antes el SVG vivía suelto debajo del `<h2>` completo) y mide su ancho real con `getBoundingClientRect()` + `ResizeObserver` (no solo al montar) — el SVG usa ese ancho medido en vez de uno fijo, así se ajusta solo a cualquier tamaño de texto/breakpoint. Verificado con Playwright: ancho SVG = ancho span exacto (155px = 155px) y mismo `left`, en desktop y en 375px.
- **Curva ondulada real, no línea recta**: `pathOndulado(ancho)` genera 2 ondulaciones vía Bezier cúbica (`C`), con los puntos de control como fracción del ancho medido (`w*0.17`, `w*0.33`, etc.) en vez de coordenadas fijas — el `viewBox` también pasó a `0 0 {ancho} 20` dinámico (antes fijo `220 20`, estirado con CSS). Verificado con Playwright leyendo el atributo `d` real del path renderizado: coordenadas escalan proporcional al ancho medido (151.9 ≈ ancho span 155px), no valores hardcodeados.

## Bug real: selección de ciudad rota en iOS (Safari/Chrome, mismo motor WebKit) — 2026-07-08

Usuario reportó en iPhone: podía abrir el buscador y tipear, pero tocar un resultado "no hacía nada" — el dropdown se quedaba abierto. No reproducible con Chromium (headless ni con touch emulado) — WebKit real no se pudo instalar en este entorno (faltan deps de sistema, requiere `sudo` interactivo).

- **Hipótesis y fix 1**: el cierre del dropdown dependía de `onBlur` + `evento.relatedTarget` (fix de accesibilidad de una sesión anterior) — en iOS ese blur a veces llega con `relatedTarget` sin setear en un tap, así que el guard no detecta que el foco se movió dentro del propio widget y cierra la lista antes de que el click llegue a disparar. Movida la selección de `onClick` a `onPointerDown` + `preventDefault()` en los botones del dropdown — ese evento dispara antes que blur en cualquier navegador/dispositivo. `onClick` se mantuvo en paralelo para activación por teclado (Enter/Espacio no disparan `pointerdown`).
- **Bug real introducido por el fix 1, encontrado verificando la seguna selección, no solo la primera**: `preventDefault()` en el botón evita que el input pierda foco al seleccionar — pero eso significa que el input queda enfocado, así que un tap posterior en el input ya no dispara `onFocus` de nuevo (el navegador solo lo dispara en una transición real de foco, y el input nunca lo había perdido). Resultado: la primera selección funcionaba, pero re-abrir el dropdown para elegir OTRA ciudad después fallaba en silencio — exactamente lo que describía el usuario. Reproducido con Playwright + touch emulado: tras seleccionar Lyon, el segundo tap en el input dejaba 0 botones en la lista.
- **Fix 2**: agregado `onClick={() => setAbierto(true)}` al input, en paralelo a `onFocus` — cubre el caso de que el input ya estuviera enfocado (donde `onFocus` no vuelve a disparar) sin romper el caso normal.
- **Verificado con Playwright, touch emulado, 3 selecciones consecutivas** (Lyon → Berlin → Marseille): las tres cambian el título correctamente, 0 errores de consola. También reverificado mouse consecutivo y teclado (Tab + Enter) en desktop, sin regresión.
- **Nota abierta**: no se pudo confirmar en WebKit real (Safari/iOS) por falta de esa dependencia en este entorno — el fix se basa en el comportamiento documentado de `pointerdown` vs `blur` en touch, no en una reproducción 1:1 del bug original. Si sigue fallando en el iPhone real tras este deploy, next step sería instalar WebKit (`sudo apt-get install` de las libs que pide `npx playwright install webkit`) para reproducir de verdad.

## Cierre de teclado + scroll suave al elegir ciudad — y bug real de fondo encontrado — 2026-07-08

Pedido: al elegir ciudad en mobile, cerrar el teclado y subir suave al tope del hero. Lo simple (blur + `window.scrollTo({top:0,behavior:'smooth'})`) saltaba de golpe en vez de animar — **investigación larga, con 4 hipótesis descartadas antes de encontrar la causa real**:

1. **`lenis.scrollTo()`**: mismo bug que el trazo de DatosDetalle (`gsap.ticker.lagSmoothing(0)`, necesario para el scrub del hero-shrink) — descartado, cambiado a scroll nativo.
2. **`lenis.stop()`/`start()` alrededor del scroll nativo** (para que Lenis no lo "peleara"): resultó ser OTRA causa de salto — el CSS propio de Lenis aplica `overflow: clip` mientras está `.lenis-stopped`, bloqueando la animación nativa igual. Descartado.
3. **Foto de la ciudad sin cachear** (descarga+decode compitiendo con el compositor): parecía correlacionar (ciudad ya vista antes no fallaba) pero agregar `Image().decode()` antes de animar no lo arregló — pista falsa, la correlación real era otra (ver 4).
4. **La causa real**: `useClimaActual`/`useCalidadAire` (`useQuery` de React Query) cambian de `queryKey` al cambiar de ciudad (`['clima', lat, lon]`) **sin `placeholderData`**, así que `data` pasa por `undefined` un instante mientras carga la ciudad nueva (si no estaba en caché — coincide exactamente con por qué "ciudad ya vista" no fallaba). `Forecast`/`DatosDetalle`/`Noticias` hacen guards tipo `if (!dias) return null`, así que ese `undefined` **desmonta las tres secciones de golpe** — la página colapsa al alto de un solo viewport (`document.documentElement.scrollHeight` medido cayendo exactamente a 844px = `clientHeight`) y el navegador clampea `scrollY` a 0 él solo, sin que ningún código JS llame `scrollTo`/`scrollIntoView`/toque `scrollTop` — confirmado interceptando las tres APIs con monkeypatching, ninguna se disparaba. Cuando el contenido vuelve a crecer, el scroll no se restaura solo.
- **Metodología para encontrarlo**: descartadas las hipótesis 1-3 desactivando cada sospechoso por separado (`useEncogimientoScroll`, el crossfade de `FondoDinamico`) sin que el bug desapareciera; confirmado que no era artefacto de Playwright probando con coordenadas de mouse crudas (sin auto-scroll-into-view) y contra producción real (no solo dev); el hallazgo definitivo fue muestrear `scrollHeight` junto a `scrollY` cada 15ms, viendo el colapso a 844px en el frame exacto del click.
- **Fix real**: `placeholderData: keepPreviousData` en ambos hooks (`useClimaActual.ts`, `useCalidadAire.ts`) — React Query sigue mostrando los datos de la ciudad anterior mientras carga la nueva, sin hueco `undefined`, sin colapso, sin clamp. Beneficio extra no pedido pero real: antes de este fix, Forecast/DatosDetalle/Noticias **parpadeaban a blanco** un instante en cada cambio de ciudad nueva (no solo el scroll se rompía) — ya no.
- Con la causa real arreglada, el código de `BuscadorCiudad.tsx` quedó simple: blur + `window.scrollTo({top:0,behavior:'smooth'})`, sin trucos. Reverificado con `scrollHeight` muestreado cada 15ms: ya no colapsa por debajo del umbral de clamp: scroll anima gradual y completo (300→287→202→...→1→0) en el flujo real, touch y mouse, ciudad cacheada y sin cachear, 3 selecciones consecutivas.
